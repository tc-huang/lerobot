# X-VLA：首個適用於任何機器人、任何任務的軟提示機器人基礎模型（Soft-Prompted Robot Foundation Model）

## 概述

多年來，機器人學一直致力於構建能夠遵循人類自然語言指令、並在多種環境和機器人本體上靈巧操作的代理（agent）。近期大型語言模型（LLM）和視覺語言模型（VLM）的突破性進展指出了一條前進道路：透過將這些基礎模型架構扎根於動作（action），將其擴展至具身控制。這催生了視覺-語言-動作（VLA）模型的興起，人們期望單一的通用模型能夠結合廣泛的語義理解與穩健的操控技能。

然而，訓練此類模型困難重重。機器人資料分散在不同的平台、感測器、具身形態和資料收集協議之間。異質性（heterogeneity）無處不在：不同的機械臂配置、不同的動作空間、不同的相機設置、不同的視覺域，以及不同的任務分佈。這些不一致性造成了嚴重的分佈偏移，使得預訓練不穩定、適配不可靠。

受元學習（meta-learning）和提示學習（prompt learning）的啟發，我們提出一個問題：**「如果 VLA 模型能夠像 LLM 透過提示學習任務一樣，學習每個機器人和資料集的結構，會怎麼樣？」**

**X-VLA** 是一個軟提示（soft-prompted）、流匹配（flow-matching）的 VLA 框架，它將每種硬體設置視為一個「任務」，並使用一小組可學習的嵌入（embedding）對其進行編碼。這些**軟提示（Soft Prompts）**捕捉了具身形態和領域特定的變異，從多模態融合的最早階段就引導 Transformer。藉由這一機制，X-VLA 能夠在單一統一架構中調和不同的機器人形態、資料類型和感測器配置。

## 訓練 X-VLA

### 建議的訓練配置

在為新的具身形態或任務微調 X-VLA 時，我們建議不要凍結 VLM，並設置 `policy.dtype=bfloat16` 以避免記憶體不足（OOM）錯誤。

```bash
lerobot-train \
  --dataset.repo_id=YOUR_DATASET \
  --output_dir=./outputs/xvla_training \
  --job_name=xvla_training \
  --policy.path="lerobot/xvla-base" \
  --policy.repo_id="HF_USER/xvla-your-robot" \
  --policy.dtype=bfloat16 \
  --policy.action_mode=auto \
  --steps=20000 \
  --policy.device=cuda \
  --policy.freeze_vision_encoder=false \
  --policy.freeze_language_encoder=false \
  --policy.train_policy_transformer=true \
  --policy.train_soft_prompts=true \
```

### 訓練參數說明

| 參數                       | 預設值  | 說明                            |
| -------------------------- | ------- | ------------------------------- |
| `freeze_vision_encoder`    | `false` | 不凍結 VLM 視覺編碼器權重       |
| `freeze_language_encoder`  | `false` | 不凍結 VLM 語言編碼器權重       |
| `train_policy_transformer` | `true`  | 允許策略 Transformer 層參與訓練 |
| `train_soft_prompts`       | `true`  | 允許軟提示參與訓練              |

**💡 最佳實踐**：對於第二階段適配新的具身形態，不要凍結 VLM 編碼器，同時訓練策略 Transformer 和軟提示。

### 範例：在雙臂機器人上訓練

```bash
lerobot-train \
  --dataset.repo_id=<USER>/bimanual-so100-handover-cube \
  --output_dir=./outputs/xvla_bimanual \
  --job_name=xvla_so101_training \
  --policy.path="lerobot/xvla-base" \
  --policy.dtype=bfloat16 \
  --policy.repo_id="YOUR_USERNAME/xvla-biso101" \
  --steps=3000 \
  --policy.device=cuda \
  --policy.action_mode=so101_bimanual \
  --policy.freeze_vision_encoder=false \
  --policy.freeze_language_encoder=false \
  --policy.train_policy_transformer=true \
  --policy.train_soft_prompts=true
```

💡 **最佳效能：** 如果您擁有足夠的運算資源並希望達到最佳的 X-VLA 微調效能，應遵循官方微調策略：

**🔥 使用自訂學習率方案全參數微調（full-finetune）所有組件**

為確保穩定最佳化，視覺語言模型（VLM）必須使用僅為基礎學習率 1/10 的學習率（LR）進行訓練，而所有其他組件使用完整的學習率（LR）。
此學習率（LR）比例對於達成強大且穩定的微調效能至關重要。預設情況下已為您自動設定。
❕注意

要完全匹配官方報告的效能，可能需要為軟提示額外設置預熱學習率排程（warm-up LR schedule），這能帶來微小的改善。
我們鼓勵在您自訂的訓練管線中實現此功能以獲得最佳結果。

## 核心概念

### 1. 動作模式

X-VLA 使用**動作註冊表（Action Registry）**系統來處理不同的動作空間和具身形態。`action_mode` 參數定義了動作的處理方式、使用的損失函數，以及預測的後處理方式。

#### 可用的動作模式

| 動作模式         | 動作維度                 | 說明                            | 使用場景             |
| ---------------- | ------------------------ | ------------------------------- | -------------------- |
| `ee6d`           | 20                       | 末端執行器含 xyz、6D 旋轉、夾爪 | 雙臂空間控制設置     |
| `joint`          | 14                       | 關節空間含夾爪                  | 直接關節控制的機器人 |
| `agibot_ee6d`    | 20                       | AGI-bot 變體，使用 MSE 損失     | AGI-bot 平台         |
| `so101_bimanual` | 20（模型）、12（實際）   | SO101 雙臂機器人                | 雙臂操控任務         |
| `auto`           | 20（模型）、自動（實際） | 自動從資料集偵測動作維度        | **推薦**用於新機器人 |

#### 動作模式的重要性

當您擁有一個以 `action_dim=20` 訓練的預訓練檢查點（如 `lerobot/xvla-base`），並希望在具有不同動作維度的資料集上訓練時（例如雙臂的 14 維），您不能簡單地裁剪動作維度。動作模式負責協調：

1. **損失計算**：針對不同動作組件使用不同的損失函數（關節用 MSE、夾爪用 BCE 等）
2. **前處理**：將夾爪通道歸零、填充維度
3. **後處理**：對夾爪 logits 應用 sigmoid、裁剪填充

#### 範例：BimanualSO101 動作空間

`so101_bimanual` 動作模式處理了模型輸出（20D）與真實機器人控制（12D）之間的不匹配：

```python
# 模型輸出 20 個維度以保持相容性
dim_action = 20

# 真實機器人只需要 12 個維度
# [左臂 (6), 右臂 (6)] = [關節 (5) + 夾爪 (1)] × 2
REAL_DIM = 12

# 前處理：將 12D 動作填充至 20D 用於訓練
# 後處理：將 20D 預測裁剪至 12D 用於部署
```

詳情請參閱 [action_hub.py](/home/jade_choghari/robot/lerobot/src/lerobot/policies/xvla/action_hub.py) 的實作。

#### Auto 的動作模式（推薦）

`auto` 動作模式是在任何機器人上使用 X-VLA 最簡單的方式。它會自動偵測您資料集的動作維度並處理填充/裁剪：

```bash
lerobot-train \
  --policy.path="lerobot/xvla-base" \
  --policy.action_mode=auto \
  --policy.max_action_dim=20 \
  ...
```

**運作方式：**

- 從您的資料集讀取 `action_feature.shape[-1]`（例如 Franka 為 7）
- 模型輸出 `max_action_dim`（預設 20）以保持預訓練相容性
- 損失**僅在實際維度上計算**：`MSE(pred[:,:,:real_dim], target[:,:,:real_dim])`
- 後處理將輸出裁剪回 `real_dim` 用於機器人控制

這使得大多數機器人無需建立自訂動作模式。

### 2. 領域 ID

領域 ID 是針對不同機器人配置和相機設置的可學習識別符。它們使 X-VLA 能夠區分：

- 不同的機器人（機器人 1 vs 機器人 2）
- 不同的相機配置（cam1 vs cam2）
- 不同的組合（Robot1-cam1-cam2 vs Robot1-cam1 vs Robot2-cam1）

#### 設置領域 ID

**訓練過程中**：預設情況下，domain_id 設為 0 用於一般訓練。

**評估過程中**：指定與您的檢查點訓練配置相匹配的 domain_id。

```python
# 範例：LIBERO 檢查點使用 domain_id=3
domain_id = 3
```

domain_id 會由前處理管線中的 `XVLAAddDomainIdProcessorStep` 自動加入觀測中。

`lerobot/xvla-base` 模型已在以下領域 ID 上進行訓練。建議選擇一個與您的機器人/配置最相似的：

### 3. 處理器的步驟

X-VLA 需要特定的前處理和後處理步驟才能正常運作。

#### 必要的前處理步驟

1. **XVLAImageToFloatProcessorStep**：將影像從 [0, 255] 轉換為 [0, 1] 範圍
2. **XVLAImageNetNormalizeProcessorStep**：應用 ImageNet 標準化（VLM 主幹網路所需）
3. **XVLAAddDomainIdProcessorStep**：將 domain_id 加入觀測中

#### 自訂處理器範例

對於 LIBERO 環境，自訂處理器處理特定的觀測格式：

```python
from lerobot.policies.xvla.processor_xvla import LiberoProcessorStep

processor = LiberoProcessorStep()
# 處理 robot_state 字典，將旋轉矩陣轉換為 6D 表示
# 應用 180° 影像旋轉以符合相機慣例
```

### 4. 配置參數

X-VLA 的關鍵配置參數：

```python
# 觀測與動作
n_obs_steps: int = 1          # 觀測時間步數
chunk_size: int = 32           # 動作序列長度
n_action_steps: int = 32       # 要執行的動作步數

# 模型架構
hidden_size: int = 1024        # Transformer 隱藏層維度
depth: int = 24                # Transformer 層數
num_heads: int = 16            # 注意力頭數
num_domains: int = 30          # 最大領域 ID 數量
len_soft_prompts: int = 32     # 軟提示嵌入長度

# 動作空間
action_mode: str = "ee6d"      # 動作空間類型（使用 "auto" 自動偵測）
use_proprio: bool = True       # 使用本體感覺狀態
max_state_dim: int = 32        # 最大狀態維度
max_action_dim: int = 20       # 填充用的最大動作維度（"auto" 模式使用）

# 視覺
num_image_views: int | None    # 相機視角數量
resize_imgs_with_padding: tuple[int, int] | None  # 帶填充的目標影像尺寸

# 訓練
num_denoising_steps: int = 10  # 流匹配去噪步數
```

## 建立自訂動作模式

如果您的機器人擁有獨特的動作空間，您可以建立自訂動作模式：

### 步驟 1：定義您的動作空間

```python
from lerobot.policies.xvla.action_hub import BaseActionSpace, register_action
import torch.nn as nn

@register_action("my_custom_robot")
class MyCustomActionSpace(BaseActionSpace):
    """我的機器人的自訂動作空間。"""

    dim_action = 15  # 您的機器人的動作維度
    gripper_idx = (7, 14)  # 夾爪通道索引

    def __init__(self):
        super().__init__()
        self.mse = nn.MSELoss()
        self.bce = nn.BCEWithLogitsLoss()

    def compute_loss(self, pred, target):
        """定義您的損失計算方式。"""
        # 範例：關節用 MSE，夾爪用 BCE
        joints_loss = self.mse(pred[:, :, :7], target[:, :, :7])
        gripper_loss = self.bce(pred[:, :, self.gripper_idx],
                                target[:, :, self.gripper_idx])

        return {
            "joints_loss": joints_loss,
            "gripper_loss": gripper_loss,
        }

    def preprocess(self, proprio, action, mode="train"):
        """訓練前的動作前處理。"""
        # 範例：將本體感覺中的夾爪歸零
        proprio_m = proprio.clone()
        action_m = action.clone() if action is not None else None
        proprio_m[..., self.gripper_idx] = 0.0
        if action_m is not None:
            action_m[..., self.gripper_idx] = 0.0
        return proprio_m, action_m

    def postprocess(self, action):
        """部署時的預測後處理。"""
        # 範例：對夾爪 logits 應用 sigmoid
        action[..., self.gripper_idx] = torch.sigmoid(action[..., self.gripper_idx])
        return action
```

### 步驟 2：使用您的自訂動作模式

```bash
lerobot-train \
  --policy.action_mode=my_custom_robot \
  --dataset.repo_id=YOUR_DATASET \
  --policy.path="lerobot/xvla-base" \
  ...
```

## 進階主題

### 多相機支援

X-VLA 透過 `num_image_views` 參數支援多相機視角：

```python
# 配置 3 個相機視角
policy.num_image_views=3

# 如果實體相機數量不足，則添加空白相機
policy.empty_cameras=1  # Adds 1 zero-padded camera view
```

### 自訂前處理 Pipeline

為您的環境建立自訂前處理 pipeline：

```python
from lerobot.processor import PolicyProcessorPipeline
from lerobot.policies.xvla.processor_xvla import (
    XVLAImageToFloatProcessorStep,
    XVLAImageNetNormalizeProcessorStep,
    XVLAAddDomainIdProcessorStep,
)

# 建立自訂 pipeline
preprocessor = PolicyProcessorPipeline(
    steps=[
        YourCustomProcessorStep(),  # 您的自訂處理
        XVLAImageToFloatProcessorStep(),  # 必要：轉換為浮點數
        XVLAImageNetNormalizeProcessorStep(),  # 必要：ImageNet 標準化
        XVLAAddDomainIdProcessorStep(domain_id=5),  # 您的領域 ID
    ]
)
```

### 處理不同的動作維度

當您的資料集動作維度少於預訓練模型時：

**選項 1（推薦）**：使用 `auto` 的動作模式

```bash
# 自動偵測您資料集的動作維度
# 適用於任何機器人，無需自訂程式碼
policy.action_mode=auto
policy.max_action_dim=20  # 匹配預訓練模型
```

**選項 2**：使用內建填充的預定義動作模式

```python
# 模型預期 20D，資料集為 12D
# 動作模式在內部處理填充
```

## 疑難排解

### 常見問題

**問題**：「動作維度不匹配（Action dimension mismatch）」

- **解決方案**：確認您的 `action_mode` 與機器人的動作空間相匹配。如有需要，建立自訂動作模式。

**問題**：「影像數值超出 [0, 1] 範圍（Image values outside [0, 1] range）」

- **解決方案**：確保影像在標準化之前已經過 `XVLAImageToFloatProcessorStep` 前處理。

**問題**：「找不到領域 ID（Domain ID not found）」

- **解決方案**：確認 `XVLAAddDomainIdProcessorStep` 在您的前處理管線中，並設定了正確的 domain_id。

**問題**：「新具身形態上的成功率低」

- **解決方案**：
  1. 確認您的 action_mode 是否正確
  2. 確認軟提示（soft prompts）是否正在訓練（`train_soft_prompts=True`）
  3. 確保正確的前處理（ImageNet 標準化、domain_id）
  4. 考慮增加訓練步數

**問題**：「訓練時記憶體不足」

- **解決方案**：
  1. 減少 `chunk_size`（例如從 32 減至 16）
  2. 啟用梯度檢查點（gradient checkpointing）
  3. 減少批次大小
  4. 凍結更多組件
