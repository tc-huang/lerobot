# 多任務 DiT Policy

多任務擴散 Transformer（DiT）策略是原始 Diffusion Policy 架構的進化版本，利用大型 DiT 搭配文本與視覺條件化來實現多任務機器人學習。此實作支援擴散（diffusion）和流匹配（flow matching）兩種目標函數來生成動作，使機器人能根據語言指令執行多樣化的操作任務。

## 模型概覽

此模型使用：

- **CLIP 視覺編碼器**：處理來自多個相機視角的 RGB 影像
- **CLIP 文本編碼器**：編碼語言任務指令（凍結權重搭配可學習的投影層）
- **擴散 Transformer**：根據觀測與語言條件預測動作序列
- **兩種目標函數**：支援擴散（DDPM/DDIM）和流匹配兩種動作生成方式

這個模型令人興奮之處在於，您僅需約 4.5 億參數就能實現極高的靈巧度，
其表現可與數十億參數的 VLA 模型競爭，且所需的訓練量顯著更少。

## 安裝需求

多任務 DiT 策略有額外的依賴套件。請使用以下指令安裝：

```bash
pip install lerobot[multi_task_dit]
```

這將安裝所有必要的依賴套件，包括用於 CLIP 模型的 HuggingFace Transformers 函式庫。

## 使用方式

若要在 LeRobot 配置中使用多任務 DiT，請將策略類型指定為：

```python
policy.type=multi_task_dit
```

## 訓練

### 基本訓練指令

以下是在您的資料集上訓練多任務 DiT 的完整訓練指令：

```bash
lerobot-train \
  --dataset.repo_id=YOUR_DATASET \
  --output_dir=./outputs/multitask_dit_training \
  --batch_size=32 \
  --steps=5000 \
  --save_freq=500 \
  --log_freq=100 \
  --policy.type=multi_task_dit \
  --policy.device=cuda \
  --policy.repo_id="HF_USER/multitask-dit-your-robot" \
  --wandb.enable=true
```

### 建議的超參數與資料集細節（30Hz 控制頻率）

為獲得可靠的效能，建議從以下預設超參數開始：

```bash
lerobot-train \
  --dataset.repo_id=YOUR_DATASET \
  --output_dir=./outputs/mutitask_dit_training \
  --batch_size=320 \
  --steps=30000 \
  --policy.type=multi_task_dit \
  --policy.device=cuda \
  --policy.horizon=32 \
  --policy.n_action_steps=24 \
  --policy.objective=diffusion \
  --policy.noise_scheduler_type=DDPM \
  --policy.num_train_timesteps=100 \
  --policy.repo_id="HF_USER/multitask-dit-your-robot" \
  --wandb.enable=true
```

**關鍵參數：**

- **Batch Size**：192-320 - 如果您有支援此規模的 GPU，將獲得最佳的訓練動態
- **Horizon**：32 - 預測的動作步數，在 30Hz 下約為 1.0 秒
- **n_action_steps**：24 - 在 30Hz 下約為 0.8 秒
- **Objective**：`diffusion` - 建議先從擴散開始，若生成品質不佳再嘗試流匹配
- **Training Steps**：單一任務建議 >30k 步

### 訓練配置參數

#### 目標函數選擇

在擴散（diffusion）和流匹配（flow matching）之間選擇：

```bash
# 擴散目標函數（預設）
--policy.objective=diffusion \
--policy.noise_scheduler_type=DDPM \  # 或 "DDIM"
--policy.num_train_timesteps=100 \
--policy.num_inference_steps=10 \  # 加速推論
--policy.beta_schedule=squaredcos_cap_v2 \  # 噪聲排程類型
--policy.prediction_type=epsilon \  # "epsilon"（預測噪聲）或 "sample"（預測乾淨樣本）
--policy.clip_sample=true \  # 在去噪過程中裁剪樣本
--policy.clip_sample_range=1.0  # 裁剪範圍 [-x, x]

# 流匹配目標函數
--policy.objective=flow_matching \
--policy.timestep_sampling_strategy=beta \  # 或 "uniform" | beta 採樣策略在實務中表現明顯更好
--policy.num_integration_steps=100 \
--policy.integration_method=euler \  # 或 "rk4"
--policy.sigma_min=0.0  # 流插值路徑中的最小噪聲
```

#### Transformer 架構

根據資料集大小調整模型容量：

```bash
# 小型資料集（< 100 個範例）
--policy.num_layers=4 \
--policy.hidden_dim=512 \
--policy.num_heads=8  # 理想情況下應為 hidden_dim // 64

# 中型資料集（100-5k 個範例）- 預設值
--policy.num_layers=6 \
--policy.hidden_dim=512 \
--policy.num_heads=8  # 理想情況下應為 hidden_dim // 64

# 大型資料集（> 5k 個範例）
--policy.num_layers=8 \
--policy.hidden_dim=512 \
--policy.num_heads=8   # 理想情況下應為 hidden_dim // 64
```

**位置編碼（Positional Encoding）選項：**

本模型支援兩種動作序列的位置編碼（positional encoding）方法：

```bash
# 旋轉位置嵌入（Rotary Position Embedding, RoPE）- 預設，建議使用
--policy.use_rope=true \
--policy.rope_base=10000.0  # RoPE 的基礎頻率

# 絕對位置編碼
--policy.use_positional_encoding=true  # 設為 true 時停用 RoPE
```

**其他 Transformer 參數：**

```bash
--policy.dropout=0.1  # DiT 區塊的 dropout 率（0.0-1.0）
--policy.timestep_embed_dim=256  # 時間步嵌入維度
```

#### 視覺編碼器配置

```bash
# 使用不同的 CLIP 模型以獲得更強的表達力，但會犧牲推論速度
# 根據任務複雜度與資料集大小，嘗試使用更大或更小的模型
--policy.vision_encoder_name=openai/clip-vit-large-patch14

# 為每個相機使用獨立的視覺編碼器
# 當相機特性差異顯著時可能有用，
# 但請注意 VRAM 佔用會增加
--policy.use_separate_rgb_encoder_per_camera=true

# 影像前處理
--policy.image_resize_shape=[XXX,YYY] \ # 您可能需要調整影像大小以加速推論
--policy.image_crop_shape=[224,224] \
--policy.image_crop_is_random=true  # 訓練時隨機裁剪，推論時中心裁剪
```

#### 文本編碼器配置

```bash
# 使用不同的 CLIP 文本編碼器模型
# 與視覺編碼器相同：根據任務複雜度與資料集大小
# 嘗試使用更大或更小的模型
--policy.text_encoder_name=openai/clip-vit-large-patch14
```

#### 學習率配置

視覺編碼器使用獨立的學習率倍率，建議以 1/10 作為理想的起始點：

```bash
--policy.optimizer_lr=2e-5 \
--policy.vision_encoder_lr_multiplier=0.1  # 視覺編碼器學習率 = 0.1 * optimizer_lr
```

### 訓練調校指南

#### 1. 搭配 Beta 採樣的流匹配（Flow Matching）

此處的原始擴散實作基於 [TRI 的 LBM 論文](https://arxiv.org/abs/2507.05331) 中描述的工作。

此外，我們實作了流匹配目標函數，其高層概念描述於 [Boston Dynamics 部落格文章](https://bostondynamics.com/blog/large-behavior-models-atlas-find-new-footing/) 中。

建議測試流匹配目標函數並評估其對您任務的效能差異：

```bash
--policy.objective=flow_matching \
--policy.timestep_sampling_strategy=beta \
--policy.timestep_sampling_alpha=1.5 \
--policy.timestep_sampling_beta=1.0 \
--policy.timestep_sampling_s=0.999
```

這並未被證明是適用於所有使用情境的銀色子彈（silver bullet），但在某些情況下確實能帶來更流暢且更一致的動作。

#### 2. Transformer 層數

根據資料集大小匹配模型容量：

- **小型資料集**（< 100 個範例）：減少至 4 層
- **大型資料集**（> 5k 個範例）：增加至 8 層

#### 3. `horizon` 調校

模型對您選擇的 horizon 可能相當敏感。建議根據控制頻率，從大約 1 秒的 horizon 開始：

- **30 Hz 頻率**：`horizon=30`
- **10 Hz 頻率**：`horizon=10`

然後嘗試從此基礎逐步增加。Horizon 決定了模型預測未來動作的時間範圍。

#### 4. `n_action_steps` 敏感度

模型對 `n_action_steps` 也可能非常敏感。建議根據控制頻率，從大約 0.8 秒開始調整：

- **較低值**：反應更靈敏，但在長時程任務中可能較不穩定
- **較高值**：長時程執行效果更好，但開迴路失敗的恢復能力受限

### 推論調校

為加速推論，使用 DDIM 搭配較少的採樣步數：

```bash
--policy.noise_scheduler_type=DDIM \
--policy.num_inference_steps=10
```

### 恢復訓練

從 checkpoint 恢復訓練：

```bash
lerobot-train \
  --config_path=./outputs/mutitask_dit_training/checkpoints/last/pretrained_model/train_config.json \
  --resume=true
```

checkpoint 目錄應包含 `model.safetensors` 和 `config.json` 檔案（訓練過程中自動儲存）。恢復訓練時，配置會從 checkpoint 載入，因此不需要指定其他參數。

## 常見失敗模式與除錯

訓練這些模型可能相當棘手。以下是常見的失敗模式與除錯方法：

### 靜止不動／無動作

模型可能在推論時「崩潰」，導致靜止或無動作輸出。這可能發生在：

1. **訓練資料不足**：如果您只有 20-50 個範例，嘗試將資料集大小大致翻倍。一旦超過 300 個範例仍然出現此問題，可能是任務過於複雜。

2. **多個相似任務**：當您的資料集包含多個相似任務（例如抓取 2 個不同物體）時，模型可能過度依賴語言條件化，而語言描述可能不夠豐富。

**除錯建議：**

- 增加資料集大小（持續翻倍直到超過 300 個範例）
- 延長訓練時間，最多至 100k 步，即使損失已趨平穩
- 檢查模型是否接收到正確的語言指令，或增加指令的多樣性

### 執行錯誤的任務

有時機器人會完全忽略您的指令而執行其他任務。這通常只在您訓練了多個任務時才會發生。

**可能原因：**

- 語言指令模糊不清
- 特定任務的訓練資料不足
- 模型在多任務資料集中混淆了相似的任務

**Debugging tips:**

- 驗證語言指令的具體性，特別是多個任務之間描述相似時
- 檢查訓練資料集中的任務分佈，並對失敗/被忽略的任務增加權重
- 考慮針對特定任務進行微調

### 訓練不穩定

如果訓練損失不穩定或發散：

- 嘗試將學習率調整在 `1e-5` 到 `3e-4` 之間
- 盡可能增加 batch size
- 檢查資料集的正規化是否正確
- 驗證影像前處理是否正常運作

## 效能考量

### GPU 需求

- **推論**：建議至少使用 RTX 5070 Ti（或同等級 GPU）以獲得合理的速度表現
- **訓練**：理想情況下需要足夠 VRAM 以支援 >64 的 batch size，具體需求視影像觀測數量等因素而異

### Batch Size 建議

- **最小值**：64（低於此值可能導致訓練不穩定）
- **建議值**：256-320（最佳效能，需要較大的 GPU）
